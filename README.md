# Геопоиск по Lat/Lng на Quadtree

Простая учебная лабораторная работа на Java: хранение точек `(lat, lng)` и поиск ближайших объектов по радиусу.

## Что здесь реализовано

- `Quadtree` — spatial index для двумерных точек
- `NaiveGeoIndex` — baseline с полным перебором
- `searchInRadius(lat, lng, radius)` — основной поиск
- `searchInBox(minLat, minLng, maxLat, maxLng)` — вспомогательный поиск по прямоугольнику
- JUnit 5 тесты
- JMH benchmark для сравнения Quadtree и полного перебора

## Почему выбран Quadtree

Quadtree хорошо подходит для такой задачи, потому что:

- карта рассматривается как обычная 2D-плоскость
- каждая вершина дерева хранит прямоугольную область
- когда точек в узле становится много, область делится на 4 квадранта
- при поиске мы идем не по всем точкам, а только по тем узлам, чьи области пересекаются с запросом

Для лабораторной это хороший компромисс: идея простая, код короткий, а на больших данных поиск обычно заметно быстрее полного перебора.

## Как устроен алгоритм

1. У корня есть фиксированная область:
   - `lat in [-90, 90]`
   - `lng in [-180, 180]`
2. Пока в узле мало точек, они лежат в обычном списке.
3. Если точек стало больше `capacity`, узел делится на 4 дочерних прямоугольника.
4. Старые точки переносятся в детей.
5. Поиск по радиусу работает в два шага:
   - сначала строится квадрат вокруг центра запроса
   - затем Quadtree возвращает кандидатов из этой области
   - после этого кандидаты фильтруются по обычному евклидову расстоянию

Формула расстояния простая, без сферической геометрии:

`distance^2 = (lat1 - lat2)^2 + (lng1 - lng2)^2`


```bash

```

### Построение индекса

| Размер данных | Naive build, ms/op | Quadtree build, ms/op |
|---:|---:|---:|
| 1 000 | 0.002 | 0.025 |
| 10 000 | 0.020 | 0.789 |

Вывод по построению:

- `NaiveGeoIndex` строится быстрее, потому что это просто добавление в список
- `Quadtree` строится медленнее, потому что при вставке есть деление узлов и перераспределение точек

| Размер данных | Радиус | Naive | Quadtree | Ускорение Quadtree |
|---:|---:|---:|---:|---:|
| 1 000 | 0.1 | 0.457 | 0.040 | ~11x |
| 10 000 | 0.1 | 4.825 | 0.062 | ~78x |
| 100 000 | 0.1 | 111.910 | 0.096 | ~1166x |
| 1 000 | 1.0 | 0.952 | 0.049 | ~19x |
| 10 000 | 1.0 | 10.979 | 0.097 | ~113x |
| 100 000 | 1.0 | 123.037 | 0.257 | ~479x |
| 1 000 | 10.0 | 1.735 | 0.158 | ~11x |
| 10 000 | 10.0 | 28.115 | 0.932 | ~30x |
| 100 000 | 10.0 | 106.711 | 9.769 | ~11x |

Вывод по поиску:

- чем больше данных, тем заметнее преимущество `Quadtree`
- чем меньше радиус, тем сильнее помогает spatial index
- при большом радиусе индекс все равно быстрее, но преимущество уменьшается
- для этой лабораторной `Quadtree` выгоден именно ускорением поиска, а не построения

## Профилинг и узкие места

### CPU

Короткий стековый профиль для `size=100000`, `radius=1.0` показал:

- у `NaiveGeoIndex` горячая точка: `NaiveGeoIndex.searchInRadius`
- у `Quadtree` горячая точка: `Quadtree.Node.search`
- это ожидаемо: baseline тратит CPU на полный проход по всем точкам, а Quadtree обходит только подходящие узлы

### Память

Короткий `gc`-профиль для `size=100000`, `radius=1.0` показал:

| Индекс | Время поиска | Выделение памяти |
|---|---:|---:|
| Naive | 110.583 us/op | 84.276 B/op |
| Quadtree | 0.223 us/op | 214.368 B/op |

Вывод по памяти:

- в текущей простой реализации `Quadtree` быстрее по CPU, но создает больше временных объектов
- главная причина в том, что `searchInRadius` сначала собирает кандидатов через `searchInBox`, а потом создает итоговый список результата
- `NaiveGeoIndex` проще по памяти, потому что делает только один линейный проход и формирует один список ответа

### Узкие места

- если радиус большой, запрос пересекает много узлов, и эффект от индекса падает
- если точки распределены очень неравномерно, отдельные ветки дерева становятся глубже других
- вставка у Quadtree дороже, чем у списка
- у текущей реализации `Quadtree` есть лишние аллокации временных списков при поиске

## Идеи улучшения

- подобрать `capacity` узла под конкретные данные
- ограничить максимальную глубину дерева
- убрать лишние временные списки в `searchInRadius` и фильтровать результат в один проход
- хранить в узлах не только точки, но и агрегированную статистику
- использовать более точную геодезическую метрику вместо плоской евклидовой
- добавить bulk loading для более быстрого построения индекса

## Что проверяют тесты

- вставку точек
- поиск по радиусу на маленьком ручном примере
- поиск по границам карты
- работу с одинаковыми координатами
- совпадение результатов `Quadtree` и `NaiveGeoIndex` на случайных данных
